发表于 2022-06-04 更新于 2022-08-09 分类于 iOS Disqus：

《Effective Objective-C 2.0》内存管理章节笔记

理解引用计数

对象创建出来时，其引用计数至少为 1。若想令其继续存活，则调用 retain 方法；若某部分代码不在使用此对象，则调用 release 或 autorelease 方法。最终当引用计数归零时，对象就回收了。

ARC

离开 if 语句就无法引用 message了，调用 NSString 的 alloc 方法返回的 message 对象的引用计数比期望值多 1，没有与之对应的释放操作来抵消，“静态分析器”很容易就能分析出这种问题。ARC 的思路正是源于此，预先加入适当的保留或释放操作来避免这些问题。

ARC 调用底层 C 语言函数如 objc_retain，能节省很多 CPU 周期。ARC 下 retain，release 和 autorelease 都不能覆写，因为它们都不被直接调用。

ARC 环境下以下列词语开头的方法，以其返回的对象归调用者所有：

  • alloc
  • new
  • copy
  • mutableCopy

调用这四种方法的那段代码要负责释放方法所返回的对象。

ARC 执行的优化

编译期优化：成对地移除能够互相抵消的 retain，release，autorelease 操作。

运行时优化

objc_autoreleaseReturnValue 检查当前方法返回之后即将要执行的那段代码，若发现要在返回的对象上执行 retain 操作，则设置全局数据结构中的一个标志位，不执行 autorelease 操作；类似的，调用方法的代码要保留对象，则执行 objc_retainAutoreleasedReturnValue。

变量的内存管理语义

  • __strong：默认，保留此值
  • __unsafe_unretained：不保留此值，可能不安全，因为再次使用变量时，其对象可能已经回收了
  • __weak：不保留此值，如果系统把对象回收，变量会自动清空
  • __autoreleasing：把对象“按引用传递”给方法时使用，此值在方法返回时自动释放？

自动释放池

自动释放池用于存放那些需要在稍后某个时刻释放的对象，清空自动释放池时，系统会向其中的对象发送 release 消息。一般情况不需要我们自己创建自动释放池块，系统会自动创建的线程，如主线程或者 GCD 机制中的线程，默认都有自动释放池。main 函数中的程序入口，也由自动释放池包裹：

释放操作等到下一次事件循环时才执行，延长对象生命期，使其在跨越方法调用边界后依然可以存活一段时间。

调用 alloc 会令 str 对象引用计数 +1，需要考虑如何将这次保留操作抵消，又不能在方法内释放 str，不然方法还没返回，系统就把该对象回收了。使用 autorelease 来释放对象，会在稍后释放对象，留下足够的时间，在需要时先保留返回值，释放操作会在清空最外层的自动释放池时执行，通常是当前线程的下一次事件循环。

嵌套自动释放池以降低内存峰值

执行 for 循环时，会有新的对象持续地创建出来，并加入自动释放池，直到 for 循环执行完才会释放，因此在执行 for 循环时，程序占用内存会持续上涨，虽然清空自动释放池时会这些临时对象会自动释放，但当循环长度无法预知时，这种情况不甚理想，例如要从数据库中读出许多对象：

僵尸对象

向已回收的对象发送消息是不安全的，启用“僵尸对象”调试功能后，运行期系统会把所有已经回收的实例转化成特殊的僵尸对象，而不会真正回收他们，僵尸对象收到消息后，会抛出异常，其中准确说明了发送过来的消息，并描述了回收之前的对象。

对某些 NSString, NSNumber 对象，把数值有关的全部消息放在指针值里面，运行期系统会在消息派发期间检测到这种标签指针。
